# Case Study - Current Vault

## 当前结构快照

截止梳理时，当前 vault 的结构特征为：
- `pages/` 共 28 页
- frontmatter 完整
- 无死链
- 无孤立页
- 读者入口和根页导航清晰

这说明当前知识库方法论本身是稳定的，真正不适合直接分享的是“本地绑定”。

---

## 从当前 vault 抽出来的通用规则

以下规则适合进入分享版模板：
- 知识库优先，不收一次性过程噪音
- 固定页面角色模型：`project / knowledge / ops / task / overview`
- 根页只做导航和稳定总览
- `ops` 页默认写成“现象 / 根因 / 处理法 / 边界”
- `log.md` 只记里程碑，不记任务回放
- 每次维护至少同步：目标页、根页、`index.md`、`log.md`
- 审计优先级：结构 > 导航 > 治理漂移 > 噪音回流

---

## 从当前 vault 必须剥离出去的本地信息

这些内容只能放进示例，不能直接进入分享版 skill 本体：
- 本地 vault 路径
- 当前 area 名称
- 当前 root page 文件名
- 当前 page map
- 本机 skills 目录路径
- 本地项目名与业务专有名词

---

## 这个案例为什么重要

它证明了一点：

> 真正值得分享的是“页面角色模型 + 降噪维护方法 + 审计顺序”，而不是某个具体 vault 的路径和命名。

因此分享版的正确抽象方式是：
- 固定 page-role model
- 把其余结构做成 `vault profile`
- 用示例案例承载本地化信息
