# Case Study - 以轻量 Harness 驱动的病理知识源导入

## 场景

团队手里有一份较长的专业 Markdown 文档，内容覆盖：
- 病理审核原则
- 标本处理与描述规则
- 常见术语与边界说明
- 操作注意事项与例外情况

这类文档通常信息很全，但直接放进知识库时会有几个典型问题：
- 单页太长，难以检索和复用
- 章节结构适合顺序阅读，不适合长期维护
- 术语、规则、操作边界混在一起
- 第一次拆分出来的页面结构，往往并不是最终最优结构

因此，这个案例的目标不是“一次性导入完成”，而是：

> 先做出一个可测试的知识库基线，再用轻量 harness 的方式，通过测试、回归和版本对比持续优化拆分方案、页面角色和链接架构。

---

## 为什么这里需要轻量 Harness

如果把长文档导入知识库看成一次性动作，常见结果是：
- 第一次拆分不合理，但后续没有稳定迭代入口
- 改完结构后，无法快速比较新旧版本差异
- 导航、来源链和 related links 改着改着就漂了
- 页面会越来越多，但结构质量并没有被持续验证

因此这里更合适的做法，不是“重型平台化”，而是把 ingest 做成一个**轻量 harness**：

- 允许快速生成一个基线版本
- 允许低成本重复运行
- 每轮都有可比较的结构产物
- 每轮调整后都有最小回归检查

它的重点不是保存所有过程日志，而是把**结构演进**变成可追踪、可比较、可复检的回路。

---

## 初始目标（Initial baseline）

第一版导入只追求**最小可用**，不追求一次到位。

预期先形成：
- 1 个 `overview` 页面：说明来源、范围、目录入口
- 若干 `chapter` 页面：承接原始章节结构
- 若干 `topic / glossary` 候选页面：收敛高频术语与概念
- 基本的 `parent / child / prev-next / related` 链接
- `toc.md`、`manifest.json`、术语候选和 related-link suggestions

这一阶段的重点不是“定稿”，而是先落一个**可跑、可测、可回退**的结构版本。

同时，这一版最好顺手产出最小 harness 产物，例如：
- `ingestion-map.md`
- `manifest.json`
- `toc.md`
- `glossary-candidates.md`
- `related-link-suggestions.md`
- `regression-checklist.md`

这样后续每轮调整时，才有稳定对照物。

---

## 第一版导入后的测试重点（Test findings）

第一版结构完成后，不直接视为最终答案，而是进入测试阶段。测试时重点观察：

1. **拆分粒度是否合理**
   - 是否有章节页过大，读者仍需长时间滚动？
   - 是否有页面过碎，导致维护成本过高？

2. **页面角色是否清楚**
   - 哪些内容其实是稳定知识，应归入 `knowledge`？
   - 哪些内容是操作边界、排查方法，应抽到 `ops`？
   - 哪些只是临时说明，不应进入长期知识库？

3. **链接结构是否真正可用**
   - 入口页能否稳定带到关键章节？
   - 术语页是否与章节页形成有效互链？
   - related links 是不是有助于回查和复用？

4. **知识库是否便于继续迭代**
   - 新版本导入后，是否方便比较前后差异？
   - 调整结构后，是否容易做回归检查？

5. **这轮产物是否适合做下次迭代输入**
   - manifest 是否足够清楚？
   - 术语候选和 related suggestions 是否有复用价值？
   - 是否已经有足够稳定的版本对比基线？

## 轻量 Harness 回路（Harness loop）

这个案例更推荐按下面的回路理解：

1. **Baseline ingest**：生成第一版知识库结构
2. **Review / test**：在实际检索、跳转、维护中暴露问题
3. **Compare**：比较当前结构与上一版的差异
4. **Refactor**：调整拆分、页面角色、链接架构
5. **Regression**：检查入口页、来源链、关键链接是否仍稳定
6. **Promote**：把这一轮稳定结果作为下一轮的基线

这个回路的关键不是“运行很多脚本”，而是：

> 让知识库结构本身具备类似 harness 的可重复验证能力。

---

## 迭代 1：调整拆分策略（Iteration 1 - Split refinement）

测试后，常见的第一类问题是：原始标题层级并不等于最佳知识库粒度。

因此第一轮通常会做：
- 把过长章节进一步按主题拆开
- 把过碎的小节重新并回章节页
- 保留原始来源链路，但不机械照搬目录层级
- 给每一页补充更清晰的页面职责说明

这一轮优化解决的是：**“能不能读”** 和 **“能不能维护”**。

---

## 迭代 2：重构页面角色（Iteration 2 - Page-role refactor）

第二类问题通常不是拆分大小，而是页面职责混乱。

常见动作包括：
- 把重复出现的重要概念提升为独立 `topic / glossary` 页面
- 把操作注意事项、排查路径、边界条件抽离到 `ops` 页面
- 把单纯的来源说明和导航保留在 `overview` 页面
- 避免让章节页同时承担知识、操作、日志三种职责

这一轮优化解决的是：**“结构是否稳定”**。

---

## 迭代 3：优化链接架构（Iteration 3 - Link architecture improvement）

第三类问题通常出现在链接层：即使页面拆得差不多了，知识仍可能“不好走”。

因此会进一步优化：
- `parent / child`：保证层级入口稳定
- `prev / next`：保留顺序阅读体验
- `related`：增强跨章节复用和横向跳转
- source lineage：保留来源可追踪性，避免重构后失去原始依据

在这一阶段，导入流程不再只是“生成页面”，而是开始真正塑造知识库的导航架构。

---

## 回归检查（Regression checks）

每轮调整后，都需要做最基本的回归，而不是改完就结束。

建议至少检查：
- overview / root page / reader entry 是否仍然可达
- 关键章节是否仍然被入口页收录
- source lineage 是否完整
- glossary / related links 是否因重构而失效
- 新结构相比上一版，是否真的更容易检索、复用和维护

如果要做得更工程化，可以把每一轮结果都视为一个版本：
- `v1`：初始导入基线
- `v2`：拆分策略优化
- `v3`：页面角色重构
- `v4`：链接架构优化与回归稳定

这样做的重点不是保存“所有过程”，而是保留**关键版本差异与结构决策**。

建议每轮至少保留一份很轻的版本记录：
- 这轮主要改了什么
- 哪些页面被拆开 / 合并 / 升格
- 哪些链接策略发生变化
- 回归检查是否通过
- 下一轮还要继续优化什么

---

## 最终稳定形态（Stable knowledge-base shape）

经过几轮测试和调整后，知识库通常会收敛到类似下面的结构：

- `overview-pathology-handbook.md`
- `knowledge-pathology-review-principles.md`
- `knowledge-pathology-specimen-rules.md`
- `knowledge-pathology-core-terms.md`
- `ops-pathology-quality-checks.md`
- `ops-pathology-boundaries.md`
- `log-pathology-ingest-iterations.md`

这里最重要的不是文件名，而是它们之间的角色分工和链接关系已经稳定下来。

---

## 为什么这个案例重要

这个案例展示的，不只是“如何把长文档拆开”，而是：

- 如何把长篇 Markdown 变成可维护的知识库结构
- 如何把第一次导入视为**可测试基线**而不是最终版本
- 如何把 ingest 流程组织成一个**轻量 harness / 回归底座**
- 如何通过测试、回归和版本对比持续优化知识架构
- 如何把一次性交付物，变成长期可复用、可演进的知识资产

一句话总结：

> `knowledge-base-ingest` 的价值，不只是拆文档，而是把长篇知识源整理成一个可以持续测试、迭代、对比和回归的知识架构演进回路。
